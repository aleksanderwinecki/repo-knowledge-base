  <action>
1. Create `src/search/field-impact.ts` with:

   **Types** (export all):
   ```typescript
   interface FieldHop {
     repoName: string;
     parentType: string;
     parentName: string;
     fieldType: string;
     nullable: boolean;
   }
   interface FieldBoundary {
     repoName: string;
     parentName: string;
     fieldType: string;
     nullable: boolean;
     topics: string[];
   }
   interface FieldImpactResult {
     fieldName: string;
     origins: FieldHop[];
     boundaries: FieldBoundary[];
     consumers: FieldHop[];
     summary: string;
   }
   interface FieldImpactCompact {
     summary: string;
     field: string;
     origins: Array<{ repo: string; schema: string; type: string; nullable: boolean }>;
     boundaries: Array<{ repo: string; proto: string; type: string; nullable: boolean; topics: string[] }>;
     consumers: Array<{ repo: string; schema: string; type: string; nullable: boolean }>;
   }
   ```

   **`analyzeFieldImpact(db, fieldName)`** algorithm:
   a. Query all field occurrences: `SELECT f.*, r.name AS repo_name FROM fields f JOIN repos r ON f.repo_id = r.id WHERE f.field_name = ? ORDER BY r.name, f.parent_type`
   b. Classify: ecto_schema -> origins, proto_message -> boundaries (potential), graphql_type -> origins (treat like ecto for display)
   c. Build service graph via `buildGraph(db)`
   d. For each boundary repo (repo with proto_message fields), find downstream repos using graph.forward: look at kafka/event edges to get consuming repo IDs and the topic/event name (from edge.via)
   e. For each consuming repo ID, check if that repo has the same field_name in its fields table. If yes, add to consumers with its local field info (parentType, parentName, fieldType, nullable)
   f. Enrich boundaries with Kafka topics: for each boundary repo, collect all topic names from graph.forward edges where mechanism is 'kafka'
   g. Build summary string: "{fieldName}: {N} origins, {M} boundaries, {K} consumers across {R} repos"
   h. Return FieldImpactResult

   **`formatFieldImpactCompact(result)`**: Convert FieldImpactResult to FieldImpactCompact with shortened field names for MCP response budget. Same 4000-char budget pattern as formatImpactCompact in impact.ts.

2. Create `tests/search/field-impact.test.ts` using the same in-memory DB test pattern as writer.test.ts:
   - Set up repos, fields, and graph edges in beforeEach

   - Use `buildGraph` + direct DB inserts to create the test scenario (2-3 repos with ecto/proto fields and kafka edges between them)
