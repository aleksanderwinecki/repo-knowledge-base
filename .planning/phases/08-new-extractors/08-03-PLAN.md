      - Per RESEARCH.md: "EventCatalog frontmatter is flat YAML; full parser is overkill"

      - Strip `event:` prefix

      - Per RESEARCH.md multi-strategy matching

   c. `enrichFromEventCatalog(db: Database, rootDir: string): { matched: number; skipped: number }`:

      - If not found, return `{ matched: 0, skipped: 0 }` silently
      - Phase 1: Parse domain MDX files at `src/domains/*/index.mdx` -- extract domain name and service IDs list
      - Phase 2: Parse service MDX files at `src/services/*/index.mdx` -- extract service ID and event IDs (sends list)

      - Phase 4: Parse event MDX files at `src/events/*/index.mdx` -- extract event ID and owners
      - Phase 5: For each catalog event, match to KB events using multi-strategy:
        1. Name match: `SELECT id FROM events WHERE name = ?` (camelCase)

        3. Path match for Payload: `SELECT id FROM events WHERE source_file LIKE ? AND name = 'Payload'` ('%' + snakeCase + '%')
      - Phase 6: UPDATE matched events: `UPDATE events SET domain = ?, owner_team = ? WHERE id = ?`
      - Wrap all UPDATEs in a single transaction (per RESEARCH.md: ~500-1000 lightweight UPDATEs, single transaction is fast and atomic)
