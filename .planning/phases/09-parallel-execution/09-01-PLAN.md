   - Returns `ExtractedRepoData` with the mode flag and all extracted data

   - The function is async (p-limit needs a Promise) but internally calls execSync for git ops -- this is fine and expected

4. Create `persistExtractedData(db, extracted)` function:

   - If `extracted.mode === 'surgical'`: call `persistSurgicalData` then `insertEventEdges`, `insertGrpcClientEdges`, `insertEctoAssociationEdges`

   - Returns `IndexStats & { mode: 'full' | 'surgical' }`
   - This function remains synchronous (better-sqlite3 is sync)

5. Refactor `indexAllRepos` to be async with three phases:
   - **Phase 1 (sequential):** Loop repos, resolve branch, run skip check, snapshot DB state (repoId + lastCommit). Build `workItems` array. This keeps ALL DB reads before the parallel phase.
   - **Phase 2 (parallel):** `const concurrency = parseInt(process.env.KB_CONCURRENCY ?? '4', 10) || 4;` then `const limit = pLimit(concurrency);`. Map workItems through `limit(() => extractRepoData(...))`. Collect with `Promise.allSettled`.
   - **Phase 3 (serial):** Loop results. For `fulfilled`: try `persistExtractedData`, log success, push to results. For `rejected` or persist error: log error, push error result. Same console.log format as current.
   - Summary line and Event Catalog enrichment unchanged from current code.
   - Return type changes to `Promise<IndexResult[]>`.

export async function withDbAsync<T>(fn: (db: Database.Database) => Promise<T>): Promise<T> {
  const db = openDatabase(getDbPath());
  try {
    return await fn(db);
  } finally {
    closeDatabase(db);
  }
}
