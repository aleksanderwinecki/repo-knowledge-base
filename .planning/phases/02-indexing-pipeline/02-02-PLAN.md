  <name>Task 1: Elixir module extractor and proto parser</name>
  <files>src/indexer/elixir.ts, src/indexer/proto.ts</files>
  <action>
Create the two domain-specific extractors. Both are pure functions returning structured data.

1. Create `src/indexer/elixir.ts`:

   Define `ElixirModule` interface:
   ```typescript
   export interface ElixirModule {
     name: string;           // "BookingContext.Commands.CreateBooking"
     type: string;           // "context" | "command" | "query" | "schema" | "module"
     filePath: string;       // relative path within repo
     moduledoc: string | null;
     functions: string[];    // ["create_booking/2", "cancel/1"]
     tableName: string | null; // for Ecto schemas only
   }
   ```

   - `extractElixirModules(repoPath: string): ElixirModule[]` -- main entry point.
   - Find all `.ex` files under `lib/` and `apps/*/lib/` (for umbrella apps). Use a simple recursive file finder -- NOT in `_build`, `deps`, `node_modules`.
   - Skip files larger than 500KB (generated code).
   - For each `.ex` file, call `parseElixirFile(repoPath, filePath, content)`.

   `parseElixirFile` logic:
   - Use regex `/defmodule\s+([\w.]+)\s+do/g` to find all module definitions.
   - For each module, determine its boundary: from `defmodule` to the next `defmodule` at the same nesting level, or end of file.
   - Extract `@moduledoc`:
     - Match `@moduledoc\s+"""([\s\S]*?)"""` for heredoc docs (use non-greedy).
     - Match `@moduledoc\s+"([^"]*)"` for single-line docs.
     - If `@moduledoc false`, set null.
     - Trim extracted doc content.
   - Extract public functions: match `def\s+(\w+)\(([^)]*)\)` within module boundary. Calculate arity from comma count in args (empty args = 0). Deduplicate (multiple clauses of same function).
   - Check for Ecto schema: match `schema\s+"(\w+)"` to get table name.
   - Classify module type:
     - If has `schema "table_name"` -> "schema"
     - If name matches `*Context` or contains `.Contexts.` or `.Context.` -> "context"
     - If name contains `.Commands.` or `.Command.` or ends with `Command` -> "command"
     - If name contains `.Queries.` or `.Query.` or ends with `Query` -> "query"
     - Otherwise -> "module"

2. Create `src/indexer/proto.ts`:

   Define `ProtoDefinition` interface:
   ```typescript
   export interface ProtoMessage {
     name: string;
     fields: { type: string; name: string }[];
   }

   export interface ProtoService {
     name: string;
     rpcs: { name: string; inputType: string; outputType: string }[];
   }

   export interface ProtoDefinition {
     filePath: string;       // relative path within repo
     packageName: string | null;
     messages: ProtoMessage[];
     services: ProtoService[];
   }
   ```

   - `extractProtoDefinitions(repoPath: string): ProtoDefinition[]` -- main entry point.
   - Find all `.proto` files recursively under the repo. Skip `node_modules`, `_build`, `deps`, `vendor`.
   - Skip files larger than 500KB.
   - For each `.proto` file, call `parseProtoFile(repoPath, filePath, content)`.

   `parseProtoFile` logic:
   - Extract `package` statement: match `^package\s+([\w.]+);` (multiline).
   - Extract messages: match `^message\s+(\w+)\s*\{` to get name. For each message, find matching closing brace (track brace depth for nested messages). Within message body, extract fields: match `^\s+(?:repeated\s+|optional\s+|required\s+)?(\w[\w.]*)\s+(\w+)\s*=\s*\d+` to get type and name.
   - Extract services: match `^service\s+(\w+)\s*\{`. Within service body, extract RPCs: match `rpc\s+(\w+)\s*\(\s*(\w[\w.]*)\s*\)\s+returns\s+\(\s*(\w[\w.]*)\s*\)` to get name, input type, output type.
   - Handle nested messages: When counting braces for message boundaries, nested message blocks are included in the parent. Extract nested messages as separate entries with dotted names (e.g., `ParentMessage.NestedMessage`).
  </action>

  <name>Task 2: Event relationship detector and pipeline orchestrator</name>
  <files>src/indexer/events.ts, src/indexer/pipeline.ts, src/index.ts</files>
  <action>
Create the event relationship detector and the main pipeline orchestrator that ties everything together.

1. Create `src/indexer/events.ts`:

   Define `EventRelationship` interface:
   ```typescript
   export interface EventRelationship {
     type: 'produces_event' | 'consumes_event';
     eventName: string;        // proto message name (e.g., "BookingCreated")
     sourceFile: string;       // file where relationship was detected
     handlerModule: string | null; // for consumers: the module that handles the event
   }
   ```

   - `detectEventRelationships(repoPath: string, protoDefinitions: ProtoDefinition[], elixirModules: ElixirModule[]): EventRelationship[]`

   Producer detection:
   - If this repo contains `.proto` files with message definitions, this repo is the producer of those events.
   - For each proto message found in this repo's proto definitions, create a `produces_event` relationship.
   - Source file = the proto file path.

   Consumer detection:
   - Scan all `.ex` files for event handler patterns:
     - `def handle_event(%EventName{` or `def handle_event(%EventName.SubType{`
     - `def handle_message(%EventName{`
     - `def handle_info({:event_name, %EventName{`
   - Use regex: `/def\s+handle_(?:event|message)\s*\(%(\w+(?:\.\w+)*)\{/gm`
   - Also check for Broadway handler patterns: `def handle_message\s*\(\s*%Broadway\.Message\{.*?data:\s*%(\w+(?:\.\w+)*)\{/s`
   - For each match, find the enclosing module name (closest preceding `defmodule`).
   - Create `consumes_event` relationship with the matched event name and handler module.

2. Create `src/indexer/pipeline.ts`:

   Define types:
   ```typescript
   export interface IndexOptions {
     force: boolean;      // bypass commit SHA check
     rootDir: string;     // root directory to scan
   }

   export interface IndexStats {
     modules: number;
     protos: number;
     events: number;
   }

   export interface IndexResult {
     repo: string;
     status: 'success' | 'skipped' | 'error';
     stats?: IndexStats;
     error?: string;
     skipReason?: string;
   }
   ```

   - `indexAllRepos(db: Database.Database, options: IndexOptions): IndexResult[]` -- main entry point.
     1. Call `discoverRepos(options.rootDir)` to find repos.
     2. For each repo, wrap in try/catch (IDX-07 error isolation):
        a. Check if skip: query repos table for this repo's `last_indexed_commit`. If commit matches current HEAD and `!options.force`, push `{ status: 'skipped', skipReason: 'no new commits' }` and continue.
        b. If stored commit exists but not reachable (force-pushed), fall back to full re-index for this repo.
        c. Call `indexSingleRepo(db, repoPath, options)`.
        d. Print status: `"Indexing {name}... done ({modules} modules, {protos} protos)"`.
     3. Print summary: total repos, indexed, skipped, errors.
     4. Return array of IndexResult.

   - `indexSingleRepo(db: Database.Database, repoPath: string, options: IndexOptions): IndexStats` -- indexes one repo.
     1. `extractMetadata(repoPath)` -- get repo metadata.
     2. Determine mode: full or incremental.
        - Full mode (force, new repo, or unreachable commit): extract everything.
        - Incremental mode: call `getChangedFiles(repoPath, lastIndexedCommit)`. Filter changed files to only relevant extensions (.ex, .proto). If no relevant changes, skip extractors.
     3. `extractElixirModules(repoPath)` -- get Elixir modules (in full mode; in incremental, only from changed files).
     4. `extractProtoDefinitions(repoPath)` -- get proto definitions.
     5. `detectEventRelationships(repoPath, protos, modules)` -- get event edges.
     6. Call `persistRepoData(db, { metadata, modules, events: protos, edges })` -- persist everything.
        - For incremental mode with deleted files: call `clearRepoFiles(db, repoId, deletedFiles)` before inserting new data.
     7. Return stats: `{ modules: modules.length, protos: protos.flatMap(p => p.messages).length, events: events.length }`.

   Note on writer integration: Plan 01's `persistRepoData` and `clearRepoFiles` need to accept the extractor types from this plan. If writer.ts needs type updates to accept ElixirModule[] and ProtoDefinition[], make those updates here. Update the `RepoData` interface in writer.ts to include the new extractor types. The writer should map ElixirModule -> modules table row, ProtoDefinition.messages -> events table rows, EventRelationship -> edges table rows.

3. Update `src/index.ts`:
   - Add exports: `extractElixirModules`, `ElixirModule` from elixir, `extractProtoDefinitions`, `ProtoDefinition`, `ProtoMessage`, `ProtoService` from proto, `detectEventRelationships`, `EventRelationship` from events, `indexAllRepos`, `indexSingleRepo`, `IndexResult`, `IndexOptions` from pipeline.
