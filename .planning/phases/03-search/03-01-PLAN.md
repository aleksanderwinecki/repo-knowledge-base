2. Create `src/search/dependencies.ts` implementing `queryDependencies()`:

Strategy:
- Resolve entity: find by name in repos table first (since edges use repo as source). If not found, check services table. Return empty result if not found.
- For direction='upstream' (what does X depend on): X's outgoing `consumes_event` edges show what it depends on. Follow to the event, then find who produces that event.
- For direction='downstream' (what depends on X): X's outgoing `produces_event` edges show what it provides. Follow to the event, then find who consumes that event.
- Use recursive CTE for multi-hop:

```sql
WITH RECURSIVE deps(entity_type, entity_id, entity_name, repo_name, mechanism, depth, path) AS (
  -- Base case: direct dependencies of target entity
  ...
  UNION ALL
  -- Recursive case: follow next hop
  ...
  WHERE depth < ? AND path NOT LIKE '%' || entity_type || ':' || entity_id || '%'
)
SELECT * FROM deps
```

- For depth='all', use max_depth=10 as safety cap
- Map relationship_type to mechanism labels:
  - 'produces_event' -> 'Kafka producer'
  - 'consumes_event' -> 'Kafka consumer'
  - 'calls_grpc' -> 'gRPC'
  - 'exposes_graphql' -> 'GraphQL'
- Build DependencyNode objects with path array showing full traversal

NOTE on edge semantics: The current pipeline stores edges as (source_type='repo', source_id=repoId, target_type='event', target_id=eventId). So for upstream: find edges WHERE source_type='repo' AND source_id=repoId AND relationship_type='consumes_event' → get target event → find edges WHERE target_type='event' AND target_id=eventId AND relationship_type='produces_event' → get source repo = the dependency. This is a 2-edge hop per dependency link.

For the recursive CTE approach, it may be cleaner to do this in TypeScript with iterative queries rather than a single CTE, given the 2-edge-per-hop semantics. Use whichever approach produces cleaner code — the key is correct results with cycle detection.

Export `queryDependencies` function.
