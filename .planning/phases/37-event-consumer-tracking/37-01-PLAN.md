<feature>
  <name>Topic-inferred consumer detection with confidence tiers</name>
  <files>src/search/field-impact.ts, tests/search/field-impact.test.ts</files>
  <behavior>
    ## New FieldConsumer type
    - Replace FieldHop[] consumers with FieldConsumer[] consumers in FieldImpactResult
    - FieldConsumer has: repoName, confidence ('inferred'|'confirmed'), via? ({topic, event}), and optional ecto fields (parentType, parentName, fieldType, nullable) only when confirmed

    ## Topic-inferred consumer detection (ECT-01, ECT-03)
    - For each boundary repo, get all proto message names (events) containing the traced field
    - For each boundary repo, get all Kafka/event forward edges (topics + consumer repo IDs) from the graph
    - Same-repo co-occurrence: attribute ALL boundary repo's proto messages to ALL its topics
    - ALL repos consuming those topics become 'inferred' consumers
    - via chain: { topic: 'the-topic', event: 'TheProtoMessage' }
    - When a boundary repo has multiple proto messages and multiple topics, each consumer gets a via entry for ONE representative topic+event pair (the first matching pair found)

    ## Confirmed consumer upgrade
    - If an inferred consumer ALSO has an ecto/graphql field matching the traced field name, upgrade to 'confirmed'
    - Confirmed consumers include both the via chain AND the ecto field details (parentType, parentName, fieldType, nullable)

    ## Dedup and self-loop exclusion
    - Use Map<number, FieldConsumer> keyed by repoId to deduplicate
    - Exclude boundary repo IDs from consumer set (existing behavior preserved)
    - Origin repo IDs should NOT be excluded -- a repo can be both an origin and a consumer via a different topic chain

    ## Backward compatibility
    - All 8 existing tests continue to pass
    - Test for 'returns consumer with nullability when downstream repo has same field' now gets confidence='confirmed' and a via chain (since it has both topic subscription AND ecto match)

    ## Test cases (RED phase targets):
    - Topic-inferred consumer (no ecto match): repo-C subscribes to topic but has no ecto field -> appears as consumer with confidence='inferred', via={topic, event}
    - Confirmed consumer (topic + ecto match): repo-B subscribes to topic AND has ecto field -> confidence='confirmed', via present, ecto fields present
    - Multiple boundaries, multiple topics: consumer of one topic gets correct via chain
    - Self-loop exclusion: boundary repo subscribing to its own topic does NOT appear as consumer
    - No consumers when no topic subscribers exist (besides boundary itself)
    - Summary string includes correct consumer count including inferred consumers
  </behavior>
  <implementation>
    1. Add FieldConsumer interface to field-impact.ts:
       ```typescript
       export interface FieldConsumer {
         repoName: string;
         confidence: 'inferred' | 'confirmed';
         via?: { topic: string; event: string };
         parentType?: string;
         parentName?: string;
         fieldType?: string;
         nullable?: boolean;
       }
       ```

    2. Update FieldImpactResult: change `consumers: FieldHop[]` to `consumers: FieldConsumer[]`

    3. In analyzeFieldImpact(), after Step 4 (building boundaries with topics):
       - Build a Map<number, FieldConsumer> (consumerMap) keyed by repoId
       - For each boundary, collect its proto message names (from boundaryMap entries sharing the same repoId)
       - For each boundary's forward edges with mechanism='kafka'|'event', get topic name (edge.via) and consumer repo IDs (edge.targetRepoId)
       - For each consumer repo ID (excluding boundary repo IDs), create an 'inferred' FieldConsumer with via={topic, event: first proto message name}
       - Use graph.repoNames to resolve consumer repo ID -> repo name

    4. In Step 5 (classify ecto fields):
       - For each ecto field in a repo that's already in consumerMap, upgrade to 'confirmed' and add ecto field details
       - For ecto fields in repos NOT in consumerMap and NOT in consumerRepoIds: classify as origins (unchanged)
       - For ecto fields in repos in the legacy consumerRepoIds set but NOT in consumerMap: also add as 'confirmed' consumers (backward compat for any edge case where graph resolution finds consumers without topic->event chain)

    5. Convert consumerMap.values() to consumers array for the result

    6. Update summary string to reflect total consumer count (inferred + confirmed)
  </implementation>
</feature>
