<feature>
  <name>Core explain module: explainService with SQL aggregation</name>
  <files>src/search/explain.ts, tests/search/explain.test.ts</files>
  <behavior>
    **Identity (EXPLAIN-02):**
    - explainService("known-repo") returns { name, description, path } from repos table
    - explainService("unknown") throws Error("Service not found: unknown")

    **Connections grouped by mechanism (EXPLAIN-02, EXPLAIN-04):**
    - Direct edges (calls_grpc, calls_http, routes_to) appear under talks_to/called_by with normalized mechanism keys (grpc, http, gateway)
    - Event-mediated connections: repo produces event consumed by other repo -> other repo appears under talks_to.event
    - Kafka-mediated connections: repo produces kafka topic consumed by other repo -> appears under talks_to.kafka
    - Inbound connections follow the reverse pattern under called_by
    - Deduplication: same (target, mechanism) pair counted once, not per-edge
    - Connections capped at ~20 per direction per mechanism group with "...and N more" truncation
    - Summary line: "Talks to 12 services (7 via grpc, 3 via event, 2 via http). Called by 5 services."
    - Summary line with zero connections: "Talks to 0 services. Called by 0 services."

    **Events (EXPLAIN-03):**
    - events.produces = names of events this repo produces (via produces_event edges)
    - events.consumes = names of events this repo consumes (via consumes_event edges)
    - Names only, no schema_definition

    **Modules (EXPLAIN-04):**
    - modules keys = module types, values = { count, top: [up to 5 names sorted alphabetically] }
    - Empty repo returns empty modules object

    **Counts (EXPLAIN-03):**
    - counts.files = file count from files table
    - counts.grpc_services = service count from services table

    **Hints (EXPLAIN-05):**
    - hints = static array with service name substituted for placeholder
    - Contains kb_impact, kb_trace, kb_deps suggestions

    **Edge cases:**
    - Repo with no edges, events, modules -> returns empty maps/arrays, summary says "Talks to 0 services. Called by 0 services."
    - Hub repo with 30+ connections -> truncated with "...and N more" entries
    - Self-referencing edges (source_id = target_id) should not appear in talks_to/called_by
  </behavior>
  <implementation>
    Create `src/search/explain.ts` with:

    1. **ExplainResult interface** as specified in RESEARCH.md recommended type.

    2. **explainService(db, name)** function:
       - Repo lookup: `SELECT id, name, path, description FROM repos WHERE name = ?` -- throw if not found
       - Direct outbound connections: query edges where source_type='repo', source_id=repo.id, target_type='repo', relationship_type IN DIRECT_EDGE_TYPES. Join repos on target_id to get target name. Normalize relationship_type to short mechanism via a local map (calls_grpc->'grpc', calls_http->'http', routes_to->'gateway').
       - Event-mediated outbound: find events this repo produces (produces_event edges), then find other repos that consume those events (consumes_event edges, source_id != this repo). Group under 'event' mechanism.
       - Kafka-mediated outbound: find kafka edges this repo produces (produces_kafka), extract topic from metadata, find other repos with consumes_kafka + same topic. Group under 'kafka' mechanism.
       - Inbound connections: reverse all three patterns (direct: target_id=repo.id; event-mediated: find events repo consumes, find other producers; kafka: find topics repo consumes, find other producers).
       - Deduplicate by (service_name, mechanism) pair within each direction.
       - Exclude self-references (same repo appearing in its own talks_to/called_by).
       - Apply ~20 cap per direction: if total unique services > 20 in a direction, truncate by removing entries from largest mechanism groups and adding "...and N more" as a special entry.
       - Build summary line using counts from the maps.
       - Events: query events table via edges (produces_event, consumes_event) for names.
       - Modules: GROUP BY type with COUNT, then top 5 names per type.
       - File count, gRPC service count.
       - Static hints array with `<this-service>` replaced by actual service name.
       - Assemble and return ExplainResult.

    3. **Helper functions:**
       - `buildConnectionSummary(talksTo, calledBy)` -> summary string
       - `truncateConnections(connections, cap)` -> truncated connections with "...and N more"
       - Mechanism normalization map: { calls_grpc: 'grpc', calls_http: 'http', routes_to: 'gateway' }

    Use the test helper patterns from tests/search/trace.test.ts (insertRepo, insertDirectEdge, insertEvent, insertEventEdge, insertKafkaEdge) for test fixtures.
  </implementation>
