**Step 1: Extend types in `src/search/types.ts`**

Add `confidence: string | null` field to `DependencyNode`. Add `mechanism?: string` field to `DependencyOptions`. The mechanism field accepts user-facing values: 'grpc', 'http', 'gateway', 'kafka', 'event'.

**Step 2: Rewrite `src/search/dependencies.ts`**

Keep the BFS structure in `queryDependencies()` intact (it handles depth, cycle detection, path tracking correctly). Changes needed:

1. Expand `MECHANISM_LABELS` to include all edge types:
   ```
   calls_grpc: 'gRPC', calls_http: 'HTTP', routes_to: 'Gateway',
   produces_kafka: 'Kafka producer', consumes_kafka: 'Kafka consumer',
   produces_event: 'Kafka producer', consumes_event: 'Kafka consumer',
   exposes_graphql: 'GraphQL'
   ```

2. Add `MECHANISM_FILTER_MAP` mapping user-facing mechanism names to relationship_type arrays:
   ```
   grpc: ['calls_grpc'], http: ['calls_http'], gateway: ['routes_to'],
   kafka: ['produces_kafka', 'consumes_kafka'], event: ['produces_event', 'consumes_event']
   ```

3. Export `VALID_MECHANISMS` as `Object.keys(MECHANISM_FILTER_MAP)` for use by CLI/MCP.

4. Pass `mechanism` option through from `queryDependencies` to `findLinkedRepos`.

5. Rewrite `findLinkedRepos()` to handle three edge patterns:

   **a) Direct edges (calls_grpc, calls_http, routes_to where target_type='repo'):**
   - Upstream: source_id = repoId, join target_id to repos table
   - Downstream: target_id = repoId, join source_id to repos table
   - These yield LinkedRepo with empty eventName, confidence from metadata

   **b) Event-mediated edges (produces_event/consumes_event):**
   - Keep existing logic (repo -> event -> repo traversal)
   - Add confidence extraction from metadata JSON (null if no metadata)

   **c) Unresolved edges (target_type='service_name', target_id=0):**
   - Query edges where source_id = repoId and target_type = 'service_name'
   - These are leaf nodes — cannot be traversed further
   - Extract targetName from metadata JSON for display
   - Include confidence from metadata

   **d) Kafka topic matching (produces_kafka/consumes_kafka):**
   - These are always unresolved (target_type='service_name', target_id=0)
   - For upstream: find topics this repo consumes_kafka, then find repos that produces_kafka the same topic
   - For downstream: find topics this repo produces_kafka, then find repos that consumes_kafka the same topic
   - Match by topic name from metadata JSON
   - This parallels event-mediated pattern but uses metadata.topic instead of event entity

6. When mechanism filter is set, restrict queries to only the relationship_types in `MECHANISM_FILTER_MAP[mechanism]`.

7. Update `LinkedRepo` interface to include `confidence: string | null` and optional `unresolvedTarget?: string`.

8. Update mechanism display strings per user decision:
   - Direct with confidence: "gRPC [high]", "HTTP [low]", "Gateway [medium]"
   - Event-mediated: "event (OrderCreated)" (confidence null for legacy)
   - Kafka: "Kafka consumer [high]", "Kafka producer [high]"
   - Unresolved: "gRPC -> [unresolved: Rpc.Partners.V1.RPCService]"

9. Update path array to include mechanism context at each hop. Format: `[repoA, "gRPC", repoB]` for direct, `[repoA, "event(OrderCreated)", repoB]` for mediated.

10. Update `queryDependencies()` to pass `confidence` through to `DependencyNode`. For unresolved leaf nodes, create DependencyNode with `type: 'unresolved'` and the unresolved target name as `name`.

11. Add `extractConfidence()` helper: parse JSON metadata, return `parsed.confidence ?? null`, catch and return null.
