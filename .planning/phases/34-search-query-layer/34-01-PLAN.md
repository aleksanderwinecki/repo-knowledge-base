<objective>
Implement OR-default search with progressive relaxation via TDD.

Purpose: AI agent search queries currently require ALL terms (implicit AND), missing relevant results. Switching to OR-default with AND->OR->prefix OR cascade dramatically improves recall while preserving precision for exact matches.

Output: `buildOrQuery()`, `buildPrefixOrQuery()`, and `searchWithRelaxation()` in fts.ts; updated `searchText()` wiring in text.ts; comprehensive tests proving correct behavior.
</objective>

  <name>OR-default queries with progressive relaxation</name>
  <files>src/db/fts.ts, src/search/text.ts, tests/db/fts.test.ts, tests/search/text.test.ts, tests/search/golden.test.ts</files>
  <behavior>
    ## buildOrQuery
    - `buildOrQuery("booking payment")` returns `"booking OR payment"` (each term tokenized individually, then joined with ` OR `)
    - `buildOrQuery("BookingCreated PaymentProcessor")` returns `"booking created OR payment processor"` (CamelCase split per term, terms joined with OR)
    - `buildOrQuery("booking")` returns `"booking"` (single term passthrough, no OR)
    - `buildOrQuery("")` returns `""` (empty input)
    - `buildOrQuery("  ")` returns `""` (whitespace-only input)
    - OR operator is NEVER passed through tokenizeForFts (would become lowercase "or")

    ## buildPrefixOrQuery
    - `buildPrefixOrQuery("booking payment")` returns `"booking* OR payment*"` (prefix wildcard per tokenized term)
    - `buildPrefixOrQuery("book")` returns `"book*"` (single term with prefix)

    ## searchWithRelaxation
    - Single-term query: runs one FTS query, no relaxation cascade
    - Multi-term AND returns >= 3 results: returns AND results (no relaxation triggered)
    - Multi-term AND returns < 3 results: retries with OR, returns OR results if >= 3
    - Multi-term AND < 3 AND OR < 3: retries with prefix OR, returns prefix OR results
    - entityTypeFilter is passed through ALL relaxation steps identically
    - limit is passed through ALL relaxation steps identically
    - Empty query returns empty array

    ## searchText integration
    - searchText now uses searchWithRelaxation instead of executeFtsQuery
    - All 15 existing golden tests still pass
    - Existing text.test.ts tests still pass (repoFilter, entityTypeFilter, limit, empty query, etc.)

    ## New golden tests
    - OR ranking: for a two-term query where both terms exist in different entities, rank #1 result contains both terms (or the most relevant match)
    - Relaxation trigger: a query whose AND returns 0-2 results returns more results after relaxation
    - Type filter preservation: relaxation with entityTypeFilter never returns results of wrong type
  </behavior>
  <implementation>
    **In src/db/fts.ts:**

    1. Add `MIN_RELAXATION_RESULTS = 3` named constant (exported for tests).

    2. Add `buildOrQuery(rawQuery: string): string` — split rawQuery on whitespace, map each term through `tokenizeForFts(term)` individually, filter empty strings, join with ` OR `. For 0-1 terms, return the single tokenized term (no OR). CRITICAL: never pass the joined OR string back through tokenizeForFts.

    3. Add `buildPrefixOrQuery(rawQuery: string): string` — same as buildOrQuery but append `*` to each tokenized term before joining with ` OR `.

    4. Add `searchWithRelaxation(db, rawQuery, limit, entityTypeFilter?): FtsMatch[]` that implements the AND->OR->prefix OR cascade:
       - Split rawQuery into terms, tokenize each individually
       - If 0 terms: return []
       - If 1 term: run single FTS query with that term, return results
       - If 2+ terms:
         - Step 1 (AND): join terms with space (implicit AND), run query. If results.length >= MIN_RELAXATION_RESULTS, return.
         - Step 2 (OR): join terms with ` OR `, run query. If results.length >= MIN_RELAXATION_RESULTS, return.
         - Step 3 (prefix OR): join terms as `term1* OR term2*`, run query. Return whatever we get.
       - Each step uses the same SQL template with entityTypeFilter and limit preserved
       - Use executeFtsWithFallback for each step, but note: OR queries constructed programmatically should never have syntax errors, so fallback is a safety net only

    5. Export `buildOrQuery`, `buildPrefixOrQuery`, `searchWithRelaxation`, and the `FtsMatch` interface (move from text.ts to fts.ts since it's now used in the fts layer).

    **In src/search/text.ts:**

    6. Remove the internal `executeFtsQuery` function entirely.

    7. Replace the FTS query call in `searchText()` with `searchWithRelaxation(db, query, limit, entityTypeFilter)` imported from fts.ts. The rest of the hydration loop stays the same — it iterates over FtsMatch results identically.

    8. Remove unused `tokenizeForFts` import from text.ts (now handled inside fts.ts).

    **In tests/db/fts.test.ts:**

    9. Add `describe('buildOrQuery')` with tests for: multi-term OR construction, single-term passthrough, empty input, CamelCase terms, OR operator survival (not lowercased).

    10. Add `describe('buildPrefixOrQuery')` with tests for: prefix wildcard appended, single term.

    11. Add `describe('searchWithRelaxation')` with tests for: single-term query, AND sufficient results, AND insufficient triggers OR, OR insufficient triggers prefix OR, entityTypeFilter preserved across steps, empty query returns [].

    **In tests/search/text.test.ts:**

    12. Add relaxation integration tests: a query that produces sparse AND results returns more results via relaxation. Type filter preserved during relaxation.

    **In tests/search/golden.test.ts:**

    13. Update golden test #4 comment (OR behavior changed — it now works correctly via programmatic OR construction).

    14. Add new golden test: "OR ranking: multi-term query ranks result with both terms highest" — search for "booking cancellation" and verify rank #1 contains both terms in name or snippet.

    15. Add new golden test: "relaxation: sparse AND query returns results via OR fallback" — search for a term combination that produces 0 AND results but has OR matches.

    16. Add new golden test: "relaxation preserves type filter" — search with entityTypeFilter and verify relaxation doesn't return wrong types.

    IMPORTANT: Do NOT change `findEntity` in entity.ts — it uses a different code path and precision matters more there (per research open question #1).
